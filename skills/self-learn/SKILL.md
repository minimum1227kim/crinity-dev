# self-learn Skill

작업 완료 후 코드베이스에서 발견한 **암묵적 규칙, 의존성 함정, 파일 구조 주의점**을 적절한 위치에 기록하여 프로젝트 전문성을 축적한다.

> 이 스킬은 인라인으로 실행한다 (별도 에이전트 없이).

---

## Step 1: 컨텍스트 수집

현재 세션에서 작업한 내용을 수집한다:

1. `git diff --name-only` 로 변경된 파일 목록 확인
2. `.claude/references/session-context.md` 읽기 (세션 컨텍스트, 존재 시)
3. 현재 대화에서 겪은 문제, 해결 방법, 발견한 패턴 회상

---

## Step 2: 발견 사항 필터링

### 추상화 원칙

발견 사항을 기록할 때 **구체적인 구현 로직이 아닌, 재사용 가능한 유형·패턴·구조**로 추상화한다. 구체적 로직은 해당 작업에서만 유효하지만, 패턴은 유사한 상황에서 반복 적용할 수 있어 학습 가치가 높다.

| 추상화 수준 | 예시 (나쁨 → 좋음) |
|------------|-------------------|
| ❌ 구체적 로직 | "`updateListConfig`에서 `params.timezone !== state.timezone` 비교 추가" |
| ✅ 패턴/구조 | "Vuex `commit`은 동기적이므로 commit 후 state 비교는 항상 동일 → commit 전에 변경 여부 저장 필수" |
| ❌ 구체적 로직 | "`reserve.js`에 `r_actions` export 추가하고 `compose.js`에서 spread" |
| ✅ 패턴/구조 | "서브모듈(reserve.js)에 새 export 타입(actions) 추가 시 상위 병합 모듈(compose.js)에도 import+spread 필요" |

**기록 전 자문**: "이 발견이 다른 모듈, 다른 기능 개발에서도 적용 가능한가?" → No이면 기록하지 않는다.

### 기록 대상 (재사용 가능한 패턴)

아래 유형 중 하나라도 해당하면 기록한다:

| 유형 | 설명 | 예시 |
|------|------|------|
| **구조 패턴** | 모듈/파일 간 의존 구조, export/import 병합 방식 | "서브모듈 export 추가 시 상위 병합 파일도 수정" |
| **상태 관리 패턴** | Store 동작 특성, 동기/비동기 타이밍 이슈 | "Vuex commit 동기성으로 인한 비교 순서 의존" |
| **데이터 변환 패턴** | 비가역 변환, 타입 불일치, 직렬화 규칙 | "epoch→문자열 변환 후 재변환 불가 → 재fetch 필수" |
| **API 계약 패턴** | FE↔BE 타입 불일치, 직렬화 형식 차이 | "Store 객체({title,value}) vs DTO(String) 매핑" |
| **초기화/순서 패턴** | 실행 순서 의존, 초기화 타이밍 제약 | "컴포넌트 mounted에서 store action 호출 필요" |
| **빌드/환경 패턴** | 환경별 차이, 도구 제약 | "Windows에서 node 경로 명시적 지정 필요" |

### 기록 제외

- `.claude/rules/` 파일에 이미 명시된 규칙
- `git log`로 파악 가능한 변경 이력
- 일반적인 프로그래밍 지식 (Java, Vue.js 일반론 등)
- 이미 CLAUDE.md `Discovered Notes`에 있는 내용
- **특정 파일의 구체적 구현 로직** (재현 가능성이 낮은 일회성 코드 변경)
- 해당 작업에서만 유효하고 다른 상황에 적용할 수 없는 발견

발견 사항이 없으면 **"새로운 발견 없음"**을 출력하고 종료한다.

## Step 3: 기록 위치 판정 + 기록

발견 사항의 성격에 따라 적합한 위치에 기록한다:

| 분류 | 기록 위치 | 예시 |
|------|----------|------|
| **범용 컨벤션/패턴** | `.claude/rules/` 해당 파일 | "approval 모듈 Entity는 `Serializable` 사용" → `backend-rules.md` |
| **보안/금지 규칙** | `.claude/rules/prohibitions.md` | "X API 호출 시 rate limit 존재" → 금지 테이블에 추가 |
| **프론트엔드 패턴** | `.claude/rules/frontend-rules.md` 또는 `frontend-ui.md` | "drive 모듈 NavigationDrawer는 별도 스크롤 영역 사용" |
| **프로젝트 고유 노트** | `CLAUDE.md` `## Discovered Notes` | "X 모듈은 Y 모듈과 순환 의존 위험" |

### rules 파일 수정 규칙

rules 파일은 프로젝트 전체에 영향을 미치므로 신중하게 다룬다:

1. **사용자 승인 필수** — rules 파일 수정 전 반드시 발견 내용과 수정 위치를 사용자에게 보여주고 승인을 받는다
2. **기존 패턴 준수** — 해당 섹션의 표 형식, 들여쓰기, 문체를 정확히 따른다
3. **출처 표시** — 새 항목 앞에 `<!-- self-learn: YYYY-MM-DD -->` HTML 주석을 달아 자동 추가 항목임을 표시한다
4. **최소 변경** — 기존 내용을 수정하지 않고, 새 항목만 추가한다

### CLAUDE.md 기록 규칙

CLAUDE.md의 `## Discovered Notes` 섹션에는 승인 없이 추가한다.

**형식**: 패턴 유형 태그를 포함하여 검색과 분류를 용이하게 한다:

```markdown
- `[영향 범위]` `패턴유형` 재사용 가능한 패턴 설명 (YYYY-MM-DD)
```

- `영향 범위`: 모듈명 또는 기술 영역 (예: `mail/store`, `Vuex`, `BE/FE API`)
- `패턴유형`: `구조`, `상태관리`, `데이터변환`, `API계약`, `초기화/순서`, `빌드/환경` 중 택 1

예시:
```markdown
- `[Vuex/store]` `상태관리` commit은 동기적으로 state를 즉시 변경 — commit 후 state 비교는 항상 동일하므로 변경 전후 비교가 필요하면 commit 전에 변수 저장 필수 (2026-03-27)
- `[mail/store/write]` `구조` 서브모듈(reserve.js)에 새 export 타입(actions) 추가 시 상위 병합 모듈(compose.js)에도 import+spread 필요 (2026-03-27)
- `[mail/store]` `데이터변환` convertMailData의 epoch→문자열 비가역 변환 — 표시 설정 변경 시 재fetch 필수 (2026-03-27)
```

**한 번에 최대 3개까지만 추가**한다 (노이즈 방지).

## Step 4: 품질 관리 + 지식 승격

기록 후 아래를 확인한다:

- **Discovered Notes 20개 초과 시**: 오래된 항목 또는 이미 rules로 승격된 항목을 사용자에게 정리 제안
- **동일 패턴유형 노트 3회 이상 반복 시**: 해당 패턴을 rules 파일로 승격할 것을 제안
- **rules로 승격된 항목**: 대응하는 Discovered Notes 항목에 `(→ rules 승격 완료)` 표시

### 지식 승격 파이프라인

발견된 패턴은 2단계로 성숙한다:

```
Discovered Notes (프로젝트 내 메모)
    ↓ 동일 패턴 반복 or 범용성 확인
Rules 파일 (프로젝트 공유 규칙)
```

#### 승격 기준

| 현재 위치 | 승격 조건 | 승격 대상 | 승격 방법 |
|----------|----------|----------|----------|
| Discovered Notes | 동일 패턴유형 3회 반복 **또는** 사용자가 승격 요청 | `.claude/rules/` 해당 파일 | 사용자 승인 후 rules 파일에 추가 |

#### 승격 판별 기준

| 판별 질문 | Yes → Rules 승격 | No → Discovered Notes 유지 |
|----------|-----------|-----------------|
| 특정 모듈명/파일명 없이 설명 가능한가? | Rules 후보 | Discovered Notes 유지 |
| 이 프로젝트의 다른 모듈에서도 적용 가능한가? | Rules 후보 | Discovered Notes 유지 |

**예시**:
- `"Vuex commit 동기성 함정"` → **Rules 승격 가능** (프로젝트 내 모든 Vuex 모듈에 적용)
- `"mail 서브모듈 export 구조"` → **Rules 승격 가능** (다른 모듈의 서브모듈에서도 동일 패턴)
- `"convertMailData 비가역 변환"` → **Discovered Notes 유지** (특정 함수에 종속)

## Step 5: 결과 보고

사용자에게 간결하게 보고한다:

```
📝 자기학습 완료
- CLAUDE.md: N개 노트 추가
- rules 승격 제안: M개 (승인 대기)
- 새로운 발견 없음 (해당 시)
```

승격 제안이 있으면 아래 형식으로 구체적 내용을 출력한다:

```
### Rules 승격 제안
| # | 패턴 | 대상 파일 |
|---|------|----------|
| 1 | {패턴 설명} | {rules 파일명} |
```

사용자가 승격을 승인하면 해당 파일을 수정하고, Discovered Notes에서 `(→ rules 승격 완료)` 표시를 업데이트한다.
