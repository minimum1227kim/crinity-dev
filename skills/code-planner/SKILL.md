# code-planner Skill

## 실행 흐름

### Step 0: 의도 구체화 (에이전트 실행 전, 스킬이 직접 수행)

**스킵 조건** — 아래 중 하나라도 해당하면 바로 Step 1로 진행:
- 사용자가 구체적인 **파일명/메서드명**을 지정한 경우
- 사용자가 **상세한 명세**를 함께 제공한 경우
- **기존 기능의 단순 수정** (버그 수정, 텍스트 변경, 설정값 변경 등)
- 사용자가 **"바로 해줘"**, **"빠르게"** 등 빠른 진행을 명시한 경우

**구체화 진행 시** (최대 3라운드, 라운드당 2~3개 질문):

| 라운드 | 질문 카테고리 | 예시 |
|--------|-------------|------|
| 1 | 범위, 핵심 동작, UI 방향 | "관리자 화면도 필요한가요?", "목록 형태 vs 카드 형태?" |
| 2 | 제약, 예외, 권한 | "기존 데이터 호환성?", "데이터 없을 때 처리?" |
| 3 | 성능, 알림, 연동 (필요 시만) | "실시간 알림이 필요한가요?" |

**구체화 완료 기준** (모두 충족 시):
- 대상 모듈이 특정됨
- 핵심 동작(CRUD, 표시, 설정 등)이 명확함
- UI 배치 방향이 대략 잡힘
- 주요 예외 케이스가 파악됨

구체화 결과를 **"정제된 요구사항"**으로 정리하여 Step 1에서 에이전트에 전달한다.

---

### Step 1: `code-planner` 에이전트 실행

정제된 요구사항(또는 원본 요청)을 전달하여 코드베이스 탐색 + 아키텍처 설계 + 테스트 시나리오 + **기능 설계 문서 생성**을 수행한다.

**사전 준비:**

1. 기능명에서 slug를 생성한다 (kebab-case 변환. 예: "주소록 그룹 관리" → `contact-group-management`)
2. 플러그인 공용 references에서 기능 설계 문서 템플릿을 Read한다:
   - 스킬 기준 상대 경로: `../../references/feature-spec-template.md`

**실행 방법:**

```
Agent tool 호출:
  - subagent_type: "crinity-dev:code-planner"
  - prompt: |
      ## 요구사항
      {정제된 요구사항 전체 텍스트 (Step 0 결과 또는 원본 요청)}

      ## 기능 설계 문서 생성 지시
      아래 템플릿을 기반으로 기능 설계 문서를 작성하여
      `.claude/references/feature-spec-{slug}.md`에 저장하라.
      slug: {slug}

      ### 템플릿 내용
      {Read한 템플릿 전체 내용}
  - description: "코드베이스 탐색 및 설계"
```

에이전트가 반환한 **계획 텍스트**(설계 결정 사항, 파일별 변경 계획, 변경 규모, 테스트 시나리오 포함)와 **기능 설계 문서 경로**를 보관한다.
이 계획 텍스트와 설계 문서는 Step 2~5 및 이후 code-developer 스킬에서 계속 참조된다.

---

### Step 2: UI 결정 필요 시 `ui-brainstorm` 스킬 실행

Step 1 에이전트 결과의 **"설계 결정 사항"** 섹션에 UI 옵션이 2개 이상 제시된 경우:

**실행 방법:**

```
Skill tool 호출:
  - skill: "crinity-dev:ui-brainstorm"
```

사용자 선택 후 결과를 계획의 "설계 결정 사항" 섹션에 반영한다.

UI 옵션이 없거나 1개만 있는 경우 이 단계를 건너뛴다.

---

### Step 3: 사용자 컨펌 대기

⏸ 다음 산출물을 사용자에게 제시하고 컨펌을 기다린다:
1. **개발 계획** (설계 결정 사항, 파일별 변경 계획, 변경 규모, 테스트 시나리오)
2. **기능 설계 문서** (`.claude/references/feature-spec-{slug}.md` — 요구사항, 화면설계, API, DB 명세)

수정 요청 시 에이전트 재실행 또는 계획/설계문서 수동 수정.

---

### Step 4: 컨펌 완료 후 처리

Step 1 에이전트 결과의 **"변경 규모"** 섹션 기준으로 복잡도 threshold 확인:

**충족 시** (수정 파일 5개 이상 / 레이아웃 변경 3개 이상 / fullstack):

```
Skill tool 호출:
  - skill: "crinity-dev:task-manager"
  - args: "create"
```

→ `.claude/tasks/{feature-slug}_task.md` 생성

태스크 분해 시 에이전트 출력의 **"파일별 변경 계획"** 레이어 컬럼을 기준으로 아래 규칙을 적용한다:

**에이전트 배정 규칙**

| 레이어 | 배정 에이전트 |
|--------|-------------|
| 백엔드 레이어 (code-planner 에이전트 출력의 "파일별 변경 계획"에서 레이어 확인) | `backend-developer` |
| 프론트엔드 레이어 (code-planner 에이전트 출력에서 레이어 확인) | `frontend-developer` |

**병렬 처리 규칙**

| 스택 유형 | 처리 방식 |
|---------|---------|
| `backend-only` | 순차: code-planner 에이전트가 결정한 레이어 순서 |
| `frontend-only` | 순차: code-planner 에이전트가 결정한 레이어 순서 |
| `fullstack` | 백엔드와 프론트엔드는 **병렬** 가능. 단, 프론트엔드 API 계층은 백엔드 API 계층 완료 후 시작 (API 계약 확인 필요) |

**미충족 시**: task-manager 호출 생략

---

### Step 5: `code-developer` 스킬로 위임

계획 수립이 완료되었으므로 개발 실행 단계로 넘어간다.

**실행 방법:**

```
Skill tool 호출:
  - skill: "crinity-dev:code-developer"
```

code-developer 스킬은 현재 대화의 계획 텍스트, 기능 설계 문서(`.claude/references/feature-spec-{slug}.md`), task.md(있는 경우)를 참조하여 개발 루프를 실행한다.
