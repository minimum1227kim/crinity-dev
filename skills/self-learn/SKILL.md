# self-learn Skill

작업 완료 후 코드베이스에서 발견한 **암묵적 규칙, 의존성 함정, 파일 구조 주의점**을 적절한 위치에 기록하여 프로젝트 전문성을 축적한다.

> 이 스킬은 인라인으로 실행한다 (별도 에이전트 없이).

---

## Step 1: 컨텍스트 수집

현재 세션에서 작업한 내용을 수집한다:

1. `git diff --name-only` 로 변경된 파일 목록 확인
2. `.claude/references/session-context.md` 읽기 (세션 컨텍스트, 존재 시)
3. 현재 대화에서 겪은 문제, 해결 방법, 발견한 패턴 회상

---

## Step 2: 발견 사항 필터링

**기록 대상** (아래 중 하나라도 해당하면 기록):
- 기존 `.claude/rules/` 파일에 **명시되지 않은** 암묵적 규칙이나 컨벤션
- 코드 구조에서 예상과 다른 의존성 관계 또는 패턴
- 디버깅 중 발견한 숨겨진 제약 조건 (예: 특정 순서 의존, 초기화 요구사항)
- 특정 모듈/파일에만 적용되는 예외적 컨벤션
- 빌드/테스트에서 발견한 환경 의존적 이슈

**기록 제외** (아래에 해당하면 기록하지 않는다):
- `.claude/rules/` 파일에 이미 명시된 규칙
- `git log`로 파악 가능한 변경 이력
- 일반적인 프로그래밍 지식 (Java, Vue.js 일반론 등)
- 이미 CLAUDE.md `Discovered Notes`에 있는 내용
- 이번 작업에서만 유효하고 재현 가능성이 낮은 일회성 이슈

발견 사항이 없으면 **"새로운 발견 없음"**을 출력하고 종료한다.

---

## Step 3: 기록 위치 판정 + 기록

발견 사항의 성격에 따라 적합한 위치에 기록한다:

| 분류 | 기록 위치 | 예시 |
|------|----------|------|
| **범용 컨벤션/패턴** | `.claude/rules/` 해당 파일 | "approval 모듈 Entity는 `Serializable` 사용" → `backend-rules.md` |
| **보안/금지 규칙** | `.claude/rules/prohibitions.md` | "X API 호출 시 rate limit 존재" → 금지 테이블에 추가 |
| **프론트엔드 패턴** | `.claude/rules/frontend-rules.md` 또는 `frontend-ui.md` | "drive 모듈 NavigationDrawer는 별도 스크롤 영역 사용" |
| **프로젝트 고유 노트** | `CLAUDE.md` `## Discovered Notes` | "X 모듈은 Y 모듈과 순환 의존 위험" |

### rules 파일 수정 규칙

rules 파일은 프로젝트 전체에 영향을 미치므로 신중하게 다룬다:

1. **사용자 승인 필수** — rules 파일 수정 전 반드시 발견 내용과 수정 위치를 사용자에게 보여주고 승인을 받는다
2. **기존 패턴 준수** — 해당 섹션의 표 형식, 들여쓰기, 문체를 정확히 따른다
3. **출처 표시** — 새 항목 앞에 `<!-- self-learn: YYYY-MM-DD -->` HTML 주석을 달아 자동 추가 항목임을 표시한다
4. **최소 변경** — 기존 내용을 수정하지 않고, 새 항목만 추가한다

### CLAUDE.md 기록 규칙

CLAUDE.md의 `## Discovered Notes` 섹션에는 승인 없이 추가한다:

```markdown
- `[모듈/파일]` 설명 (YYYY-MM-DD)
```

예시:
```markdown
- `[mail/entity]` CrMailAttach는 companyId 없이 mailId로만 조회 가능 — 부모 CrMail을 통해 테넌트 격리 (2026-03-26)
- `[approval]` Entity 상속 패턴이 Serializable — 다른 모듈(Entity 인터페이스)과 다름 (2026-03-26)
```

**한 번에 최대 3개까지만 추가**한다 (노이즈 방지).

---

## Step 4: 품질 관리

기록 후 아래를 확인한다:

- **Discovered Notes 20개 초과 시**: 오래된 항목 또는 이미 rules로 승격된 항목을 사용자에게 정리 제안
- **동일 주제 노트 3회 이상 반복 시**: 해당 패턴을 rules 파일로 승격할 것을 제안
- **rules 파일에 추가한 항목**: 대응하는 Discovered Notes 항목이 있으면 `(→ rules 승격 완료)` 표시 후 다음 정리 시 삭제 대상으로 마킹

---

## Step 5: 결과 보고

사용자에게 간결하게 보고한다:

```
📝 자기학습 완료
- CLAUDE.md: N개 노트 추가
- rules 수정 제안: M개 (승인 대기)
- 새로운 발견 없음 (해당 시)
```

rules 수정 제안이 있으면 구체적 내용을 함께 출력하여 사용자가 바로 판단할 수 있도록 한다.
